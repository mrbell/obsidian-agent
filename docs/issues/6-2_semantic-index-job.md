# 6-2 — Semantic Index Job (Embedding Phase)

**Status**: `open`
**Parent**: 6
**Depends on**: 6-1

## Description

Implement the embedding phase of `obsidian-agent index-semantic`. This phase is pure Python
(no LLM, no network after model download): it chunks changed notes, generates embeddings
using the local model, and stores results in DuckDB. Fast enough to run daily.

## Incremental Logic

New module: `obsidian_agent/index/semantic.py`

A note needs re-embedding when:
- It exists in `notes` but has no rows in `chunks` (new note)
- Its `content_sha256` has changed since any of its chunks were last embedded
  (compare `notes.content_sha256` against what was used at embed time)

**Approach**: Store the `content_sha256` that was current when embedding ran. Simplest
implementation: add `embedded_sha256 TEXT` to the `chunks` table alongside `embedded_at`.
If `notes.content_sha256 != chunks.embedded_sha256` for any chunk of that note, the whole
note is stale.

**On note change**: Delete all existing chunks and chunk_embeddings for that note, rechunk
and re-embed, insert fresh rows.

**On note deletion**: Chunks are deleted via cascade from the `notes` table delete
(add `FOREIGN KEY` or handle explicitly in the scan step). Also delete associated rows in
`chunk_concepts`, `chunk_entities`, `implicit_items`. Delete `note_intelligence` row.

**On note rename**: The existing `index` build step already updates `note_relpath` in place
across tables. `chunks` and `chunk_embeddings` should follow the same rename logic: update
`note_relpath` (and `chunk_id` prefix) in the semantic tables rather than re-embedding.

## Algorithm

```
1. Load all notes from DB: {note_relpath -> content_sha256}
2. Load all chunk metadata: {note_relpath -> embedded_sha256 (first chunk)}
3. Stale notes = notes where embedded_sha256 is missing or != content_sha256
4. For each stale note (batched):
   a. Read note content from vault filesystem
   b. Chunk using chunker.py
   c. Embed all chunks in one batch call to LocalEmbedder
   d. In a transaction:
      - Delete old chunks, chunk_embeddings for this note
      - Insert new chunks (with embedded_sha256 = current sha256)
      - Insert chunk_embeddings
      - Delete note_intelligence row (stale; will be regenerated by 6-3)
5. Log stats: notes processed, chunks generated, chunks embedded
```

**Batching**: Embed all chunks for a note in one `embedder.embed([...])` call. Optionally
batch multiple notes together if the local model benefits from larger batches (typically yes).

## CLI

```
obsidian-agent index-semantic [--config PATH] [--verbose]
```

Runs both the embedding phase (this issue) and the intelligence phase (6-3). The two phases
are sequential within the same command. The embedding phase always runs first so chunk IDs
exist before concept extraction references them.

Add to `cli.py` as a new top-level command. No business logic in `cli.py` — delegate to
`obsidian_agent/index/semantic.py`.

## Cron Integration

```
# Daily: structural index first, then semantic enrichment
0 2 * * * obsidian-agent index && obsidian-agent index-semantic
```

Run structural `index` first so `notes` table is current before the semantic step checks
for stale notes.

## Tests

- `test_semantic_index.py`: use a temp vault + temp DuckDB; verify that:
  - New notes get chunked and embedded
  - Unchanged notes are skipped on second run
  - Changed notes (updated sha256) are re-embedded
  - Deleted note chunks are removed
  - `note_intelligence` is cleared for re-embedded notes

Mock `LocalEmbedder` to return fixed-size random vectors without loading the model.

## Definition of Done

- `obsidian-agent index-semantic` runs on a real vault, embedding phase only, without error
- Incremental logic verified: second run with no changes processes 0 notes
- Changing a note causes that note's chunks to be replaced on next run
- Stats logged (notes processed, chunks generated)
